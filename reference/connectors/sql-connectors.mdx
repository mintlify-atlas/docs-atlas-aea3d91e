---
title: "SQL connectors"
description: "Connect to relational databases using SQLAlchemy-backed SQL connectors."
---

The `SQLConnector` is Meerschaum's primary connector type. It connects to relational databases via [SQLAlchemy](https://www.sqlalchemy.org/) and can serve as both a fetch connector (data source) and an instance connector (data storage).

## Built-in connectors

Three `sql` connectors are pre-configured out of the box:

| Connector | Description | URI |
|-----------|-------------|-----|
| `sql:main` | The pre-configured [TimescaleDB](https://github.com/timescale/timescaledb) instance included in the [Meerschaum Stack](/reference/stack/). Runs on `localhost` and is shared across environments. | `postgresql+psycopg://mrsm:mrsm@localhost:5432/meerschaum` |
| `sql:local` | A SQLite file inside the Meerschaum root directory. Isolated per environment. | `sqlite:///$MRSM_ROOT_DIR/sqlite/mrsm_local.db` |
| `sql:memory` | An in-memory SQLite database. Not persistent; isolated per process. | `sqlite:///:memory:` |

## Supported database flavors

The following flavors are confirmed feature-complete through the Meerschaum test suite, listed in descending order of compatibility and performance:

<AccordionGroup>
  <Accordion title="PostgreSQL-based">
    - **TimescaleDB** — recommended for time-series workloads
    - **PostgreSQL**
    - **Citus** — distributed PostgreSQL
    - **PostGIS** — PostgreSQL with geospatial extensions
    - **CockroachDB**
  </Accordion>
  <Accordion title="Embedded databases">
    - **SQLite** — zero-configuration file-based database
    - **DuckDB** — analytical in-process database
    - **GeoPackage** — SQLite with spatial extensions
  </Accordion>
  <Accordion title="MySQL-compatible">
    - **MySQL 5.7+**
    - **MariaDB**
  </Accordion>
  <Accordion title="Enterprise databases">
    - **Microsoft SQL Server** (requires `pyodbc`)
    - **Oracle SQL** (requires `oracledb`)
  </Accordion>
</AccordionGroup>

## Configuration

Connectors can be configured using individual attributes or a URI string.

<Tabs>
  <Tab title="Attributes">
    ```yaml
    meerschaum:
      connectors:
        sql:
          mydb:
            flavor: postgresql
            host: localhost
            port: 5432
            database: mydb
            username: myuser
            password: mypassword
    ```
  </Tab>
  <Tab title="URI">
    ```yaml
    meerschaum:
      connectors:
        sql:
          mydb:
            uri: postgresql://myuser:mypassword@localhost:5432/mydb
    ```
  </Tab>
  <Tab title="SQLite">
    ```yaml
    meerschaum:
      connectors:
        sql:
          local:
            flavor: sqlite
            database: /path/to/database.db
    ```
  </Tab>
  <Tab title="DuckDB">
    ```yaml
    meerschaum:
      connectors:
        sql:
          duck:
            flavor: duckdb
            database: /path/to/database.duckdb
    ```
  </Tab>
</Tabs>

### URI formats by flavor

| Flavor | URI prefix |
|--------|------------|
| TimescaleDB | `timescaledb://` or `postgresql+psycopg://` |
| PostgreSQL | `postgresql://` or `postgresql+psycopg://` |
| SQLite | `sqlite:////path/to/file.db` |
| DuckDB | `duckdb:///path/to/file.duckdb` |
| MySQL | `mysql+pymysql://user:pass@host/db` |
| MariaDB | `mariadb+pymysql://user:pass@host/db` |
| MSSQL | `mssql+pyodbc://user:pass@host/db?driver=ODBC+Driver+17+for+SQL+Server` |
| Oracle | `oracle+oracledb://user:pass@host/service` |

<Note>
  URIs beginning with `postgres://` are automatically rewritten to `postgresql+psycopg://`. URIs beginning with `timescaledb://` set the flavor to `timescaledb` and route through the psycopg driver.
</Note>

## Creating a connector from a URI in Python

```python
import meerschaum as mrsm

# Create from URI directly
conn = mrsm.get_connector(
    type='sql',
    label='mydb',
    uri='postgresql://myuser:mypassword@localhost:5432/mydb',
)

# Parse a URI into its components
from meerschaum.connectors.sql import SQLConnector
params = SQLConnector.parse_uri('sqlite:////home/foo/bar.db')
# {'database': '/home/foo/bar.db', 'flavor': 'sqlite'}
```

## In-place syncs

When a pipe's fetch connector and instance connector are the same `sql` connector, Meerschaum performs the sync entirely in SQL — no data is loaded into RAM. This is the recommended approach for incrementally materializing views.

```python
import meerschaum as mrsm

weather_pipe = mrsm.Pipe(
    'plugin:noaa', 'weather', 'atl',
    columns={
        'datetime': 'timestamp',
        'id': 'station',
    },
    parameters={
        'noaa': {
            'stations': ['KATL'],
        },
    },
)

inplace_pipe = mrsm.Pipe(
    'sql:main', 'weather_avg', 'atl',
    columns={
        'datetime': 'day',
        'station': 'station',
    },
    parameters={
        'sql': f"""
            SELECT
                TIME_BUCKET('1 day', timestamp) AS "day",
                station,
                AVG("temperature (degC)") AS avg_temp
            FROM "{weather_pipe.target}"
            GROUP BY "day", station
        """,
    },
)

# Both connectors are sql:main — sync happens entirely in SQL.
success, msg = inplace_pipe.sync()
```

## SQLConnector utility methods

### `read()`

Pass a table name or SQL query to fetch a Pandas DataFrame:

```python
import meerschaum as mrsm

conn = mrsm.get_connector('sql:main')
df = conn.read('SELECT 1 AS foo')
# df:
#    foo
# 0    1
```

Use `chunksize` and `as_iterator=True` to stream large result sets via server-side cursors:

```python
table = 'sql_main_weather_avg_atl'
chunks = conn.read(table, chunksize=25, as_iterator=True)
for chunk in chunks:
    print(f"Loaded {len(chunk)} rows.")
# Loaded 25 rows.
# Loaded 25 rows.
# Loaded 13 rows.
```

### `to_sql()`

Persist a DataFrame to a table (wraps `pandas.DataFrame.to_sql()`):

```python
import meerschaum as mrsm
import pandas as pd

conn = mrsm.get_connector('sql:main')
df = pd.DataFrame([{'a': 1, 'b': 2}, {'a': 3, 'b': 4}])
results = conn.to_sql(df, 'foo', as_dict=True)
mrsm.pprint(results)
# {
#     'target': 'foo',
#     'num_rows': 2,
#     'duration': 0.09040146699408069,
#     'success': True,
#     'msg': 'It took 0.09 seconds to sync 2 rows to foo.'
# }
```

### `exec()`

Execute a SQL statement and return the SQLAlchemy result. Useful for DDL and DML queries:

```python
import meerschaum as mrsm

conn = mrsm.get_connector('sql:main')
_ = conn.exec('DROP TABLE IF EXISTS foo')
_ = conn.exec('CREATE TABLE foo (bar INT)')

result = conn.exec('INSERT INTO foo (bar) VALUES (1), (2)')
print(f"Inserted {result.rowcount} rows.")
# Inserted 2 rows.
```

### `exec_queries()`

Execute multiple queries as a transaction. Set `break_on_error=True` to roll back on failure:

```python
import meerschaum as mrsm
from meerschaum.utils.sql import table_exists

conn = mrsm.get_connector('sql:main')
conn.exec('DROP TABLE IF EXISTS foo')

# This transaction fails on the second query and rolls back.
queries = [
    'CREATE TABLE foo (bar INT)',
    'CREATE TABLE foo (a INT, b INT)',  # duplicate table — will fail
]
results = conn.exec_queries(queries, break_on_error=True, silent=True)
success = len(results) == len(queries)
assert not success

# The table was not created because the transaction rolled back.
assert not table_exists('foo', conn)
```

## `meerschaum.utils.sql` helper functions

### `build_where()`

Build a parameterized `WHERE` clause from a `params` dictionary:

```python
from meerschaum.utils.sql import build_where
import meerschaum as mrsm

conn = mrsm.get_connector('sql:main')
print(build_where({'foo': [1, 2, 3]}, conn))
# WHERE
#     "foo" IN ('1', '2', '3')

# Prefix a value with '_' to negate it:
print(build_where({'foo': ['_3', '_4']}, conn))
# WHERE
#     ("foo" NOT IN ('3', '4'))
```

### `table_exists()`

Check whether a table exists:

```python
import pandas as pd
import meerschaum as mrsm
from meerschaum.utils.sql import table_exists

conn = mrsm.get_connector('sql:main')
df = pd.DataFrame([{'a': 1}])
conn.to_sql(df, 'foo')

print(table_exists('foo', conn))  # True
conn.exec('DROP TABLE foo')
print(table_exists('foo', conn))  # False
```

### `get_sqlalchemy_table()`

Return a SQLAlchemy `Table` object for use with the ORM or Core:

```python
import pandas as pd
import meerschaum as mrsm
from meerschaum.utils.sql import get_sqlalchemy_table

conn = mrsm.get_connector('sql:main')
df = pd.DataFrame([{'a': 1}])
conn.to_sql(df, 'foo')

table = get_sqlalchemy_table('foo', conn)
print(table.columns)
# {'a': Column('a', BIGINT(), table=<foo>)}
```

### `get_table_cols_types()`

Return a dictionary mapping column names to their SQL data types:

```python
from meerschaum.utils.sql import get_table_cols_types
import meerschaum as mrsm
import pandas as pd

conn = mrsm.get_connector('sql:main')
df = pd.DataFrame([{'a': 1}])
conn.to_sql(df, 'foo')

cols_types = get_table_cols_types('foo', conn)
print(cols_types)
# {'a': 'BIGINT'}
```

### `wrap_query_with_cte()`

Wrap a subquery in a CTE and append an enclosing query — useful for flavors like MSSQL that do not support nested CTEs:

```python
from meerschaum.utils.sql import wrap_query_with_cte

sub_query = "WITH foo AS (SELECT 1 AS val) SELECT (val * 2) AS newval FROM foo"
parent_query = "SELECT newval * 3 FROM src"
query = wrap_query_with_cte(sub_query, parent_query, 'mssql')
print(query)
# WITH foo AS (SELECT 1 AS val),
# [src] AS (
#     SELECT (val * 2) AS newval FROM foo
# )
# SELECT newval * 3 FROM src
```
