Meerschaum's plugin system is designed to be simple so you can get your plugins working quickly. Plugins are Python modules placed in the Meerschaum configuration `plugins` directory and are imported at startup under the global namespace `plugins`.

## Creating a plugin

<Steps>
  <Step title="Navigate to your plugins directory">
    <Tabs>
      <Tab title="Linux / macOS">
        ```bash
        cd ~/.config/meerschaum/plugins
        ```
      </Tab>
      <Tab title="Windows">
        ```
        %APPDATA%\Meerschaum\plugins
        ```
      </Tab>
      <Tab title="Meerschaum Compose">
        ```
        plugins/
        ```
        (relative to your `mrsm-compose.yaml`)
      </Tab>
    </Tabs>

    <Note>
      If you have set `MRSM_ROOT_DIR` or `MRSM_PLUGINS_DIR`, navigate to your designated plugins directory instead.
    </Note>
  </Step>

  <Step title="Create your Python module">
    Create either `<name>.py` for a single-file plugin or `<name>/__init__.py` for a package.

    Plugins may have hyphenated names, e.g. `mongodb-connector`.
  </Step>

  <Step title="Define your version string (optional)">
    ```python
    # ~/.config/meerschaum/plugins/example.py
    __version__ = '0.0.1'
    ```

    <Info>
      To publish changes to a repository with `register plugin`, you must increment `__version__` according to [SemVer](https://semver.org/). You may omit `__version__` for the initial release, but subsequent releases need it defined.
    </Info>
  </Step>

  <Step title="Declare your dependencies (optional)">
    List package requirements in the module-level `required` list. These will be installed into the plugin's virtual environment at installation time or with `mrsm install required <plugin>`.

    ```python
    required = ['rich>=9.8.0', 'pandas', 'plugin:foo']
    ```

    <Tip>
      Dependencies that start with `plugin:` are Meerschaum plugins. To require a plugin from another repository, add the repository's keys after the plugin name, e.g. `plugin:foo@api:bar`.
    </Tip>
  </Step>

  <Step title="Write your functions">
    The special functions are `fetch()`, `sync()`, `register()`, `setup()`, and a function matching the plugin name. Use decorators like `@make_action`, `@make_connector`, `@api_plugin`, `@dash_plugin`, `@web_page`, `@pre_sync_hook`, and `@post_sync_hook` for more capabilities.

    <Warning>
      You must include `**kwargs` in every plugin function to capture optional arguments. The `fetch()`, `sync()`, and `register()` functions also require a `pipe` positional argument.
    </Warning>
  </Step>
</Steps>

## Performance tip

For optimal performance, keep module-level code to a minimum — especially heavy imports.

<CodeGroup>

```python bad.py
# BAD — module-level import
import pandas as pd

def fetch(pipe, **kw):
    return pd.read_csv('data.csv')
```

```python good.py
# GOOD — import inside the function
def fetch(pipe, **kw):
    import pandas as pd
    return pd.read_csv('data.csv')
```

</CodeGroup>

## Special functions

### `register(pipe, **kwargs)`

The `register()` function returns a new pipe's parameters dictionary. It is called once when a pipe is first registered.

```python
def register(pipe: 'meerschaum.Pipe') -> dict:
    """
    Prompt the user for stations when registering new pipes.
    """
    stations = ask_for_stations(pipe)
    return {
        'columns': {'datetime': 'timestamp', 'id': 'station'},
        'noaa': {'stations': stations},
    }
```

<Note>
  If you are using Meerschaum Compose, `register()` is overridden by `mrsm-compose.yaml` and may be skipped.
</Note>

### `fetch(pipe, **kwargs)`

The fastest way to leverage Meerschaum's syncing engine. Return a DataFrame, list of dictionaries, or a chunk generator. Meerschaum handles filtering duplicates and updating your tables.

```python
# ~/.config/meerschaum/plugins/example.py

__version__ = '0.0.1'
required = ['requests']

def register(pipe, **kw):
    return {
        'columns': {
            'datetime': 'dt',
            'id': 'id',
        },
    }

def fetch(pipe, **kw):
    import requests, datetime, random
    response = requests.get('http://date.jsontest.com/')
    data = response.json()
    timestamp = datetime.datetime.fromtimestamp(
        int(str(data['milliseconds_since_epoch'])[:-3])
    )
    return [{
        'dt': timestamp,
        'id': random.randint(1, 4),
        'value': random.uniform(1, 100),
    }]
```

For large datasets, return a generator of DataFrame-like chunks:

```python
from typing import Generator, List, Dict, Any

def fetch(pipe, **kw) -> Generator[List[Dict[str, Any]], None, None]:
    return (
        [
            {'id': i, 'val': 10.0 * i},
            {'id': i + 1, 'val': 20.0 * i},
        ]
        for i in range(10)
    )
```

You can also yield chunks from a CSV reader:

```python
def fetch(pipe, **kw):
    import pandas as pd
    return pd.read_csv('very-large.csv', chunksize=10_000)
```

### `sync(pipe, **kwargs)`

The `sync()` function overrides the built-in syncing process and returns a `SuccessTuple` (`Tuple[bool, str]`). Use this when you need finer-grained control, such as syncing multiple pipes or doing custom multiprocessing.

```python
from __future__ import annotations
from typing import Tuple, Any, Optional, Dict
import datetime

__version__ = '0.0.1'
required = ['requests', 'pandas']

def sync(
        pipe: 'meerschaum.Pipe',
        begin: Optional[datetime.datetime] = None,
        end: Optional[datetime.datetime] = None,
        params: Dict[str, Any] = None,
        **kw: Any
    ) -> Tuple[bool, str]:
    import pandas as pd
    import requests
    import meerschaum as mrsm

    url = 'https://api.example.com/json'
    params = params or {}
    if begin is not None:
        params['begin'] = begin.isoformat()
    if end is not None:
        params['end'] = end.isoformat()

    try:
        df = pd.read_json(requests.get(url, params=params).text)
    except Exception as e:
        return False, f"Failed to sync data from {url}: {e}"

    success, msg = pipe.sync(df)
    if not success:
        return success, msg

    another_pipe = mrsm.Pipe('foo', 'bar', instance='sql:local')
    return another_pipe.sync(df)
```

### `@make_connector`

Define a new connector type by creating a class that inherits from `meerschaum.connectors.Connector` and decorating it with `@make_connector`.

```python
# plugins/foo.py

from datetime import datetime
from typing import List, Dict, Any, Optional
import meerschaum as mrsm
from meerschaum.connectors import make_connector, Connector

required = ['requests']

@make_connector
class FooConnector(Connector):

    REQUIRED_ATTRIBUTES = ['username', 'password']

    def fetch(
        self,
        pipe: mrsm.Pipe,
        begin: Optional[datetime] = None,
        end: Optional[datetime] = None,
        **kwargs: Any
    ) -> List[Dict[str, Any]]:
        params = {}
        if begin:
            params['begin'] = begin.isoformat()
        if end:
            params['end'] = end.isoformat()
        response = self.session.get('https://foo.com/data', params=params)
        return response.json()

    @property
    def session(self) -> 'requests.Session':
        _sesh = self.__dict__.get('_session', None)
        if _sesh is not None:
            return _sesh
        import requests
        self._session = requests.Session()
        self._session.auth = (self.username, self.password)
        return self._session
```

Register new connectors interactively with `mrsm bootstrap connector`, or define them as environment variables:

```bash
export MRSM_FOO_BAR='{
    "username": "bar",
    "password": "fuzz"
}'
```

<Note>
  To designate your connector as an instance connector, add `IS_INSTANCE = True` to the class. Add `IS_THREAD_SAFE = True` to enable multithreading.
</Note>

### `@make_action`

Add custom commands to the `mrsm` CLI. Actions integrate into the larger Meerschaum system and can run as background jobs or via the web console.

```python
# ~/.config/meerschaum/plugins/sing.py

from meerschaum.plugins import make_action

@make_action
def sing_song(**kw):
    return True, "This action is called 'sing song'."

@make_action(daemon=False)
def sing_tune(**kw):
    return True, "The action `sing tune` will not run in the CLI daemon context."

def sing(**kw):
    """
    Functions with the same name as the plugin are considered actions.
    """
    return True, 'Hello, World!'
```

<Tip>
  Actions with underscores become sub-actions: `foo_bar` is the same as `mrsm foo bar`.
</Tip>

You can also provide shell auto-completions with a `complete_<action>()` function:

```python
@make_action
def foo_bar(**kw):
    return True, "Success"

def complete_foo_bar(**kw):
    return ['option 1', 'option 2']
```

### `@api_plugin`

Extend the Meerschaum web API by accessing the FastAPI app. The initialization function is executed on `mrsm start api`.

```python
# ~/.config/meerschaum/plugins/example.py

from meerschaum.plugins import api_plugin

@api_plugin
def init_plugin(app):
    """
    Executed immediately after the app is initialized.
    """
    import fastapi
    from meerschaum.api import manager

    @app.get('/my/new/path')
    def new_path(
        curr_user=fastapi.Depends(manager)
    ):
        return {'message': f'The current user is {curr_user.name}'}
```

<Tip>
  Navigate to `https://localhost:8000/docs` to test your endpoint in the Swagger UI.
</Tip>

### `@dash_plugin` and `@web_page`

Add [Plotly Dash](https://dash.plotly.com/) pages to the Web Console.

```python
# ~/.config/meerschaum/plugins/example.py

from meerschaum.plugins import dash_plugin, web_page

@dash_plugin
def init_dash(dash_app):

    import dash.html as html
    import dash_bootstrap_components as dbc
    from dash import Input, Output, no_update

    # Routes to '/dash/my-page'
    @web_page('/my-page', login_required=False)
    def my_page():
        return dbc.Container([
            html.H1('Hello, World!'),
            dbc.Button('Click me', id='my-button'),
            html.Div(id='my-output-div'),
        ])

    @dash_app.callback(
        Output('my-output-div', 'children'),
        Input('my-button', 'n_clicks'),
    )
    def my_button_click(n_clicks):
        if not n_clicks:
            return no_update
        return html.P(f'You clicked {n_clicks} times!')
```

<Note>
  `@web_page` may be used with or without arguments. When invoked without arguments, it uses the function name as the path. Sub-paths are routed to your base callback, e.g. `/items/apple` is accepted by `@web_page('/items')`.
</Note>

### `@pre_sync_hook` and `@post_sync_hook`

Inject callbacks that fire before and after the `sync pipes` action.

```python
from datetime import datetime
import meerschaum as mrsm
from meerschaum.plugins import pre_sync_hook, post_sync_hook

@pre_sync_hook
@post_sync_hook
def log_sync(
        pipe: mrsm.Pipe,
        success_tuple: mrsm.SuccessTuple | None = None,
        sync_timestamp: datetime | None = None,
        sync_complete_timestamp: datetime | None = None,
        sync_duration: float | None = None,
        **kwargs
    ) -> mrsm.SuccessTuple:
    if success_tuple is None:
        print(f'About to sync {pipe} at {sync_timestamp}...')
        return True, 'Success'

    success, message = success_tuple
    prefix = 'successfully' if success else 'failed to'
    msg = (
        f'It took {sync_duration:.2f}s to {prefix} sync {pipe}\n'
        + f'({sync_timestamp} - {sync_complete_timestamp})'
    )
    return True, msg
```

Keyword arguments available to `@post_sync_hook` callbacks:

| Argument | Type | Description |
|---|---|---|
| `success_tuple` | `SuccessTuple` | The return value of the sync method |
| `sync_complete_timestamp` | `datetime` | UTC datetime after the sync completed |
| `sync_duration` | `float` | Duration of the sync in seconds |

Additional context arguments available to both hooks:

| Argument | Type | Description |
|---|---|---|
| `sync_method` | `Callable` | Bound function of `pipe.sync()`, `pipe.verify()`, or `pipe.deduplicate()` |
| `sync_timestamp` | `datetime` | UTC datetime captured before the sync began |
| `name` | `str` | Background job name (if spawned with `--name`) |
| `schedule` | `str` | Job cadence (if spawned with `-s`) |
| `begin` / `end` | `datetime \| int` | Values for `--begin` and `--end` |

### `setup(**kwargs)`

Executed during plugin installation or with `mrsm setup plugins <plugin>`. Return a `SuccessTuple`.

```python
def setup(**kw):
    global geckodriver_location
    try:
        from webdriver_manager.firefox import GeckoDriverManager
        geckodriver_location = GeckoDriverManager().install()
    except Exception as e:
        return False, str(e)
    return True, 'Success'
```

## Keyword arguments

Many useful CLI flags are passed to plugin functions as keyword arguments.

<Tip>
  Inspect `**kwargs` by running `mrsm show arguments` to see all available values.
</Tip>

<AccordionGroup>
  <Accordion title="Common keyword arguments">
    | Argument | CLI flag | Type | Description |
    |---|---|---|---|
    | `action` | Positional | `Optional[List[str]]` | Positional arguments following the action name |
    | `yes` | `-y`, `--yes` | `bool` | When `True`, yes/no prompts default to yes |
    | `force` | `-f`, `--force` | `bool` | Implies `-y`; skip confirmation prompts |
    | `loop` | `--loop` | `bool` | Action should run continuously |
    | `begin` | `--begin` | `Optional[datetime]` | Start datetime for date-bounded queries |
    | `end` | `--end` | `Optional[datetime]` | End datetime for date-bounded queries |
    | `connector_keys` | `-c`, `--connector-keys` | `Optional[List[str]]` | Filter pipes by connector |
    | `metric_keys` | `-m`, `--metric-keys` | `Optional[List[str]]` | Filter pipes by metric |
    | `location_keys` | `-l`, `--location-keys` | `Optional[List[str]]` | Filter pipes by location |
    | `mrsm_instance` | `-i`, `--instance` | `Optional[str]` | Connector keys of the Meerschaum instance |
    | `repository` | `-r`, `--repository` | `Optional[str]` | Connector keys of the plugin repository |
  </Accordion>
  <Accordion title="Edge case keyword arguments">
    | Argument | Type | Description |
    |---|---|---|
    | `sysargs` | `List[str]` | Corresponds to `sys.argv[1:]` |
    | `line` | `Optional[str]` | `sysargs` joined by spaces (shell only) |
    | `sub_args` | `Optional[List[str]]` | Items in `sysargs` enclosed in square brackets |
    | `filtered_sysargs` | `List[str]` | `sysargs` without `sub_args` |
    | `shell` | `bool` | `True` when executed from the Meerschaum shell |
  </Accordion>
</AccordionGroup>

### Custom CLI flags

Add your own arguments with `add_plugin_argument()`. It takes the same parameters as `argparse.add_argument()`:

```python
# ~/.config/meerschaum/plugins/example.py

from meerschaum.plugins import add_plugin_argument
add_plugin_argument('--foo', type=int, help='This is my help text!')
```

This produces the following in `mrsm --help`:

```
Plugin 'example' options:
  --foo FOO             This is my help text!
```

## Working with plugins programmatically

### Import another plugin

```python
# ~/.config/meerschaum/plugins/foo.py
import meerschaum as mrsm
bar = mrsm.Plugin('bar').module
```

### Activate a plugin in a REPL

```python
import meerschaum as mrsm
plugin = mrsm.Plugin('foo')
plugin.activate_venv()
foo = plugin.module
```

### Use a plugin in a script

```python
from meerschaum.utils.venv import Venv
from meerschaum import Plugin

plugin = Plugin('foo')
with Venv(plugin):
    foo = plugin.module
```

## Publishing as a Python package

If you want to manage your plugin as a proper Python package (e.g. to publish to PyPI), add the `meerschaum.plugins` entry point to your package metadata:

<Tabs>
  <Tab title="pyproject.toml">
    ```toml
    [project.entry-points."meerschaum.plugins"]
    foo = "foo"
    ```
  </Tab>
  <Tab title="setup.py">
    ```python
    from setuptools import setup

    setup(
        ...,
        entry_points={
            'meerschaum.plugins': [
                'foo = foo',
            ],
        },
    )
    ```
  </Tab>
</Tabs>
